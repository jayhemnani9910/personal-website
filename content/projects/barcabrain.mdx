  context: |
    Traditional player scouting relies on manually filtering databases by stats like goals, assists, or tackles. But football is multi-dimensional—a box-to-box midfielder might excel at passing, pressing, and dribbling simultaneously. Vector search solves this by encoding each player as a point in high-dimensional space where **similar playing styles cluster together naturally**.

    Instead of asking "Who scored 15+ goals last season?", you can ask "Who plays most like Kevin De Bruyne?" and get players with similar pass creativity, vision, and positioning—even if their raw stats differ. This is powerful for:
    - **Like-for-like replacements**: Find affordable players who mirror your departing star
    - **Tactical fits**: Identify players whose style matches your formation needs
    - **Hidden gems**: Discover undervalued talent in lower leagues with similar profiles to elite players

  architecture: |
    The system follows a classic embedding-based retrieval pipeline:

    **1. Feature Engineering** → Transform raw match events (passes, shots, tackles) and player metadata (position, age, league) into 50-100 numerical features capturing playing style.

    **2. Embedding Generation** → Use dimensionality reduction (PCA/UMAP) or learned embeddings to compress features into dense 16-32 dimensional vectors that preserve similarity relationships.

    **3. Vector Index** → Store embeddings in a specialized data structure (FAISS, Annoy, or Qdrant) optimized for fast nearest-neighbor search across thousands of players.

    **4. Query & Filter** → Given a player, retrieve top-k most similar embeddings, apply hybrid filters (position, league, age range), and rank by cosine similarity.

    **5. Insights UI** → Visualize results with similarity scores, attribute comparison radar charts, and cluster maps showing player positioning in tactical space.

  keyDecisions:
    - decision: "FAISS with IndexFlatIP for exact search"
      rationale: "Dataset is relatively small (5k-10k players). Exact search ensures no false negatives, and query latency remains <100ms. Avoided approximate indexes (IVF) since accuracy mattered more than speed at this scale."

    - decision: "Feature engineering over pre-trained models"
      rationale: "No robust pre-trained embeddings exist for football player styles. Custom feature engineering from event data gave full control over what 'similarity' means—emphasizing tactical actions over just goals/assists."

    - decision: "Dimensionality reduction to 24 dimensions"
      rationale: "Reduced from 80 raw features using PCA. Balances compression (faster search, less storage) with preservation of variance (95% retained). Sweet spot for visualization and performance."

    - decision: "Hybrid filtering (vector + metadata)"
      rationale: "Pure vector search might return strikers when querying for midfielders. Adding position/league filters post-retrieval ensures results are tactically valid while preserving similarity ranking."

        import faiss
        import numpy as np

        # Build FAISS index
        dimension = 24
        index = faiss.IndexFlatIP(dimension)  # Inner product (cosine similarity)

        # Normalize embeddings to unit length for cosine similarity
        faiss.normalize_L2(embeddings)
        index.add(embeddings)  # Add all player embeddings

        # Query: Find players similar to "Pedri" (index 42)
        query_embedding = embeddings[42:43]  # Shape: (1, 24)
        faiss.normalize_L2(query_embedding)

        k = 10  # Top 10 similar players
        similarities, indices = index.search(query_embedding, k)

        # Results
        for i, (idx, score) in enumerate(zip(indices[0], similarities[0])):
