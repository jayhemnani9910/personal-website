        # Embed user query into same vector space as stored chunks

        # PostgreSQL query using pgvector's cosine distance operator (<=>)

            SELECT content, summary, screenshot_path,
                   1 - (embedding <=> %s) as similarity
            FROM text_chunks

            LIMIT 10

        # Processing screenshots through OCR to searchable text
        import pytesseract
        from PIL import Image

        def process_screenshot(screenshot_path):
            # Load screenshot and extract text with Tesseract
            image = Image.open(screenshot_path)
            extracted_text = pytesseract.image_to_string(
                image,
                config='--psm 6'  # Assume uniform block of text
            )

            # LLM generates contextual summary and tags
            summary = llm_summarize(extracted_text)
            tags = llm_extract_tags(extracted_text)

            # Chunk text into semantic units (paragraphs, code blocks)
            chunks = semantic_chunker(extracted_text)

            # Generate embeddings for each chunk
            embeddings = model.encode(chunks)

            # Store in PostgreSQL with vector embeddings
            store_chunks(chunks, embeddings, summary, tags)
      explanation: "The processing pipeline transforms raw screenshots into searchable knowledge. Tesseract extracts text, LLMs add understanding (what's this about?), and chunking creates retrievable units. The key insight: we're not just storing pixels or OCR text, but semantic meaning that enables 'search by concept' - find that React component you saw last week by describing what it did, not remembering exact variable names."

    - "Semantic search with embeddings is transformative for personal memory because you rarely remember exact keywords - you remember concepts and context. Vector similarity lets you find 'that authentication code I saw' without remembering if it was labeled 'login', 'auth', or 'session'."

    - "The 'good enough' threshold for OCR is lower than you'd think - 90% accuracy works fine for semantic search because embeddings capture meaning even with some character errors. Perfect transcription matters less than capturing key concepts and context."
    - "Storage optimization matters at scale - 1K screenshots is multiple GB. Smart compression and deduplication (don't capture identical screens) keeps the system sustainable for long-term use without filling up drives."
    - "Chunking strategy significantly impacts retrieval quality. Too large = irrelevant content dilutes relevance score. Too small = lose context. Semantic chunking (paragraph boundaries, code blocks) works better than fixed-size windows."
    - "Vector search threshold tuning is critical - too strict (>0.9 similarity) returns nothing, too loose (<0.5) returns garbage. The 0.7-0.8 range works well for most queries, but exposing this as a user parameter helps with different use cases."

# Overview
Production personal knowledge assistant combining screenshot capture, optical character recognition, and LLM-powered semantic search. Automatically processes 1K+ screenshots with OCR, embeds 100K+ text chunks into vector database, and retrieves relevant context in under 200ms. Eliminates manual organization by building searchable memory from screen activity with zero-effort capture.
