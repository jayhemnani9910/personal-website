  context: |
    The speedcubing community relies on precise timing and detailed statistics to track improvement and prepare for competitions. Existing timer applications at the time had several pain points: inconsistent timing accuracy due to JavaScript event loop delays, limited statistical insights beyond basic averages, and no visual feedback for scramble verification.

    Speedcubers need more than just a stopwatch. They require:
    - Sub-millisecond accuracy for competitive timing
    - Rolling averages (Ao5, Ao12, Ao100) to track consistency
    - Session management to separate practice types (OH, 2x2, 3x3, etc.)
    - Scramble generation following WCA (World Cube Association) standards
    - Visual verification of scrambles to ensure proper randomization

    This project aimed to create a browser-based timer that matches the reliability of dedicated hardware timers while adding features only possible in software: comprehensive statistics, session history, and 3D scramble visualization.

  architecture: |
    The application follows a modular architecture with three core subsystems:

    **Timing Engine**
    - High-resolution timer using performance.now() API for microsecond precision
    - Event-driven state machine (IDLE → READY → RUNNING → STOPPED)
    - Jitter compensation layer to handle JavaScript event loop delays
    - Spacebar detection with press-and-hold mechanism (WCA standard behavior)

    **Statistics Module**
    - Real-time computation of rolling averages (Ao5, Ao12, Ao100)
    - Best/worst solve tracking with DNF (Did Not Finish) handling
    - Session persistence using localStorage with JSON serialization
    - Statistical outlier detection for data quality

    **3D Visualization**
    - WebGL-based cube renderer with camera controls
    - Scramble parser that converts WCA notation to 3D transformations
    - Animation system for smooth rotation transitions
    - Fallback to Canvas 2D for older browsers

    All modules communicate through a central event bus, ensuring loose coupling and testability.

  components: |
    **TimerDisplay Component**
    - Large, high-contrast display showing current time
    - Color states: red (not ready), green (ready), blue (running)
    - Handles keyboard input with precise timing capture
    - Inspection timer mode (15-second countdown for official solves)

    **ScrambleGenerator**
    - Random-state scramble generation (not just random moves)
    - WCA-compliant notation (F, B, U, D, L, R with modifiers)
    - Validation to ensure scrambles don't cancel out
    - Copy-to-clipboard functionality for sharing

    **StatsPanel**
    - Live-updating statistics grid (current, best, session average)
    - Time list with solve-by-solve breakdown
    - Plus-two penalty and DNF marking
    - Graph view showing solve time trends over session

    **CubeRenderer (WebGL)**
    - 26 cubelet model (center piece is hidden)
    - Quaternion-based rotation for smooth animations
    - Mouse/touch controls for camera manipulation
    - Sticker colors matching standard WCA color scheme

    **SessionManager**
    - Multiple session support (tabs or dropdown)
    - Import/export functionality (JSON format)
    - Session comparison view for tracking improvement
    - Auto-save with conflict resolution

  dataFlow: |
    **Timing Flow:**
    1. User presses spacebar → State transitions to READY (after 300ms hold)
    2. User releases spacebar → Timer starts, performance.now() captures t0
    3. Timer updates every 16ms (60 FPS) for smooth display
    4. User presses spacebar again → Timer stops, captures t1, computes delta
    5. Result published to event bus → Statistics module and UI update

    **Scramble Flow:**
    1. ScrambleGenerator creates random-state scramble on timer stop
    2. Scramble string parsed into move array (e.g., ["R", "U'", "F2"])
    3. CubeRenderer receives moves and applies transformations
    4. Animation queue executes rotations sequentially
    5. Final state validated against expected configuration

    **Statistics Flow:**
    1. New solve time added to session array
    2. Rolling averages recalculated (trimmed means for Ao5/Ao12)
    3. Personal best checked and updated if beaten
    4. Session data serialized and saved to localStorage
    5. StatsPanel receives update event and re-renders

    **State Persistence:**
    - On every solve completion, session state written to localStorage
    - On app load, previous session restored from localStorage
    - Export creates downloadable JSON with full session history
    - Import validates JSON schema before merging with current data

  keyDecisions: |
    **1. Using performance.now() Over Date.now()**
    Date.now() has millisecond precision and can be affected by system clock changes. performance.now() provides microsecond precision and is monotonic, ensuring timing accuracy even if the user adjusts their system clock mid-solve.

    **2. Jitter Compensation Strategy**
    JavaScript timers aren't guaranteed to fire exactly on time due to event loop delays. Implemented a compensation algorithm that measures the delta between expected and actual callback time, adjusting the displayed time accordingly. This reduced timing variance from ±50ms to ±2ms.

    **3. WebGL for 3D Rendering**
    Initially considered Three.js for easier development, but opted for raw WebGL to minimize bundle size (critical for mobile users). The custom renderer is ~8KB vs ~500KB for Three.js, with sufficient features for a simple cube model.

    **4. Rolling Average Implementation**
    WCA averages trim the best and worst solves before calculating the mean. This required careful handling of edge cases: what if the session has fewer than 5 solves? What if multiple solves are DNF? Implemented a robust algorithm that handles all cases per WCA regulations.

    **5. localStorage vs IndexedDB**
    Chose localStorage for simplicity, with JSON serialization of session data. IndexedDB would offer better performance for very large datasets, but testing showed localStorage handles 1000+ solves without noticeable lag. Trade-off favored simpler code and better browser compatibility.

    **6. Spacebar Hold Mechanism**
    WCA regulations require a ready state to prevent accidental timer starts. Implemented a 300ms press-and-hold detection before the timer becomes ready (green state). This matches the behavior of professional speedcubing timers and feels natural to experienced cubers.

  learnings: |
    **Technical Insights:**

    *Timing Precision in JavaScript*
    Learned that browser timing APIs have surprising quirks. Performance.now() precision varies by browser (Firefox gives microseconds, Chrome rounds to 5μs for security). Discovered that requestAnimationFrame provides smoother updates than setInterval for display refresh.

    *WebGL Programming Model*
    First deep dive into WebGL shader programming. Understanding the vertex/fragment shader pipeline was challenging but rewarding. Key insight: move as much computation to the GPU as possible - calculating colors per-vertex rather than per-fragment improved frame rates significantly.

    *State Machine Design*
    Implementing the timer as an explicit state machine (IDLE → READY → RUNNING → STOPPED) eliminated entire classes of bugs. Previously had implicit state scattered across variables, leading to race conditions when users pressed keys rapidly.

    **UX Lessons:**

    *Feedback is Critical*
    Early versions had subtle state changes that users missed. Adding color coding (red/green/blue) and sound effects dramatically improved the experience. Learned that multimodal feedback (visual + audio) prevents user errors.

    *Mobile Performance Challenges*
    Initial WebGL implementation ran at 60 FPS on desktop but <20 FPS on mobile. Optimization lessons: reduce draw calls, use simpler shaders, disable anti-aliasing on low-end devices. Implemented automatic quality adjustment based on measured frame rate.

    *Data Visualization Matters*
    Adding the solve-time graph was an afterthought, but users loved it. Seeing progress visually is more motivating than just numbers. Learned to invest in data presentation, not just data collection.

    **Community Feedback:**

    *Feature Requests vs. Simplicity*
    Speedcubers requested many features (blind-solving mode, cross-timer, move counters). Learned to balance feature requests with maintaining a clean, focused UX. Added most-requested features but kept them hidden behind settings to avoid cluttering the main interface.

    *Standards Compliance*
    Initially used a simpler scramble generation algorithm (random moves). Community feedback pushed me to implement proper random-state scrambles that match WCA standards. Learned the value of following established standards even when they're more complex.

    **What I'd Do Differently:**

    If rebuilding today, I'd use TypeScript for better type safety around timing logic and state transitions. Would also consider using Web Workers for scramble generation to avoid blocking the main thread. Finally, would implement proper unit tests from the start - timing bugs are subtle and hard to catch manually.
