  keyDecisions:
    - decision: "TimescaleDB over Cassandra"
      rationale: "Chose TimescaleDB because 90% of queries involve time-range filters and aggregations, which TimescaleDB optimizes via automatic chunking and compression. Standard SQL interface reduced learning curve for analysts familiar with PostgreSQL. Cassandra would have required denormalized tables and custom aggregation logic."
    - decision: "Kafka for streaming, not direct DB writes"
      rationale: "Kafka provides replay capability for debugging, decouples producers from consumers for independent scaling, and enables multiple downstream consumers (e.g., alerting, archival) without modifying the ingestion pipeline. Buffering in Kafka prevents backpressure on market data feeds during database slowdowns."
    - decision: "Star schema for analytics, not wide denormalized tables"
      rationale: "Star schema enables analysts to join on different dimensions (sector analysis, exchange comparisons) without replicating symbol metadata in every row. Materialized views cache frequent joins while preserving schema flexibility for new dimensions."
    - decision: "Airflow for orchestration, not cron"
      rationale: "Airflow's DAG dependencies ensure dimensional loads complete before fact inserts, preventing orphaned records. Retry logic and alerting are built-in. UI provides visibility into backfill progress and data lineage across 20+ daily DAGs."

        from kafka import KafkaProducer
        import json

        producer = KafkaProducer(
            bootstrap_servers=['localhost:9092'],
            value_serializer=lambda v: json.dumps(v).encode('utf-8'),
            acks='all',  # Wait for all replicas
            retries=5,   # Retry on transient failures
            enable_idempotence=True  # Prevent duplicates on retry
        )

        def publish_tick(symbol, price, volume, timestamp):
            message = {
                'symbol': symbol,
                'price': price,
                'volume': volume,
                'timestamp': timestamp,
                'dedup_key': f"{symbol}_{timestamp}"  # For DB upsert
            }
            producer.send('market-ticks', value=message, key=symbol.encode())

    - title: "TimescaleDB Continuous Aggregate for Minute Candles"

        -- Create hypertable for tick data
        CREATE TABLE ticks (
            timestamp TIMESTAMPTZ NOT NULL,
            symbol_id INT NOT NULL,
            price NUMERIC(10,2),
            volume INT,
            dedup_key TEXT UNIQUE
        );
        SELECT create_hypertable('ticks', 'timestamp');

        -- Continuous aggregate: real-time minute candles
        CREATE MATERIALIZED VIEW minute_candles
        WITH (timescaledb.continuous) AS
        SELECT
            time_bucket('1 minute', timestamp) AS bucket,
            symbol_id,
            first(price, timestamp) AS open,
            max(price) AS high,
            min(price) AS low,
            last(price, timestamp) AS close,
            sum(volume) AS volume
        FROM ticks
        GROUP BY bucket, symbol_id;

        -- Refresh policy: update every 30 seconds
        SELECT add_continuous_aggregate_policy('minute_candles',
            start_offset => INTERVAL '1 hour',
            end_offset => INTERVAL '1 minute',
            schedule_interval => INTERVAL '30 seconds');

# Overview
Production-grade platform unifying batch and streaming market data pipelines into a single analytical warehouse. Combines Kafka-based real-time tick ingestion (10K+ events/second) with Airflow-orchestrated batch OHLC loads, storing everything in TimescaleDB hypertables optimized for time-series queries. Star schema dimensional modeling enables slice-and-dice analytics across symbols, exchanges, and sectors. Interactive Dash dashboards provide minute-level candles, rolling technical indicators (SMA, EMA), and intraday heatmaps with sub-2-second load times for multi-month histories.

The platform solves the classic data engineering challenge: market data arrives through multiple channels (REST APIs for historical OHLC, WebSocket feeds for real-time ticks) with different schemas and latency requirements. Rather than maintaining separate systems, this unified architecture enables analysts to query both real-time and historical data through a single SQL interface, with reproducible backfills and clear data lineage for all derived metrics.
